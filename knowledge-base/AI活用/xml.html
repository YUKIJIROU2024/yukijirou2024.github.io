読むより、見てつかむXML

XMLは「データに意味を付けるための箱」

タグで囲むことで、データの意味・階層・関係を人にもコンピューターにも伝えます。HTMLのように画面を作るものではなく、JSONのようにデータ交換にも使われる、厳密な構造化フォーマットです。

自由なタグ 親子構造 テキスト形式 システム間連携 AIプロンプト整理にも便利

最初に結論

XML = データを「意味のあるタグ」で包み、階層構造として保存・交換する形式。

たとえば <氏名>山田太郎</氏名> と書けば、単なる文字列ではなく「これは氏名だ」と伝えられます。

データ交換 設定ファイル Office内部構造 AI指示の整理
1

XMLは画面表示ではなく、データ構造

HTMLは見た目、XMLは意味と構造を表す役割が中心です。

2

タグは自由に作れる

<price>でも<価格>でも、目的に合わせて設計できます。

3

ルールは厳密

開始タグと終了タグ、入れ子、ルート要素、大文字小文字に注意します。

全体像：XMLは「箱の中に箱を入れる」

XML文書は、1つの大きなルート要素の中に、子要素や属性を配置して作ります。

基本文法：XMLを構成する部品

📌

XML宣言

<?xml version="1.0" encoding="UTF-8"?>
文書の種類と文字コードを示します。

🏷️

要素

<title>吾輩は猫である</title>
開始タグ・値・終了タグのまとまり。

🔖

属性

<book id="001">
要素に付ける補足情報です。

🌳

階層

親要素の中に子要素を入れ、データの構造を表します。

<?xml version="1.0" encoding="UTF-8"?>
<bookstore>
  <book id="001">
    <title>吾輩は猫である</title>
    <author>夏目漱石</author>
    <price currency="JPY">600</price>
  </book>
</bookstore>
💡
属性と要素の使い分け属性は「ID・区分・単位」のような補足情報に向きます。本文として扱いたい情報は要素にするほうが読みやすくなります。

絶対に守るべきルール

XMLは自由にタグを作れますが、構文は厳密です。少しのズレで読み込みエラーになります。

1. 大文字・小文字を区別する

OK <Data>中身</Data>

NG <Data>中身</data>

2. 入れ子は正しく閉じる

OK <親><子>中身</子></親>

NG <親><子>中身</親></子>

3. ルート要素は1つだけ

文書全体を包む一番外側のタグは1つ必要です。

4. 空要素は省略形が使える

<改行 /> のように、中身がないタグは1つにまとめられます。

よくあるエラー：特殊文字

XML本文に < や & をそのまま書くと、タグや記号として解釈されることがあります。次のようにエスケープします。

書きたい文字XMLでの書き方意味
<&lt;小なり
>&gt;大なり
&&amp;アンパサンド
"&quot;ダブルクォート
'&apos;シングルクォート

XML・HTML・JSONの違い

形式主な目的書き方向いている場面注意点
XML データ交換・保管・構造化 自由なタグで囲む 厳密な階層、属性、文書構造が必要な場面 記述量が多くなりやすい
HTML Webページ表示 決まったタグを使う 画面・文章・リンク・レイアウトの表示 データ交換の主役ではない
JSON 軽量なデータ通信 キーと値のペア Web API、JavaScript連携、軽量通信 コメントや属性表現は苦手
🎯
覚え方HTMLは「見せる」、JSONは「軽く渡す」、XMLは「意味と構造を厳密に渡す」。

AIプロンプト

指示・前提・入力・出力形式を分離し、AIの理解を助けます。

AIプロンプトでXMLタグを使う

AIにとってXMLタグは、命令・材料・制約・出力形式を見分ける「区切り線」のように働きます。

メリット

  • 指示と対象テキストを分離できる
  • 長いプロンプトでも見通しが良くなる
  • 制約事項を見落とされにくい
  • 出力フォーマットを指定しやすい

日本語タグでOK

<指示>、<前提条件>、<対象テキスト> のような日本語タグでも使えます。

英語の文書を処理する時だけ、英語タグにそろえると安定する場合があります。

<前提条件>
あなたはプロのマーケターです。
親しみやすく、信頼感のあるトーンで書いてください。
</前提条件>

<指示>
以下の対象テキストを読み、20代の社会人に響くキャッチコピーを3つ作成してください。
</指示>

<対象テキスト>
仕事帰りに通える、初心者向けの定額制英会話スクール。
オンライン受講も可能。
</対象テキスト>

<出力形式>
・キャッチコピー1
・キャッチコピー2
・キャッチコピー3
</出力形式>
<タスク>
以下の会議議事録を読み、決定事項と次回アクションを整理してください。
</タスク>

<制約事項>
・箇条書きで簡潔にまとめること。
・推測は含めず、議事録に記載されている事実のみを抽出すること。
</制約事項>

<会議議事録>
Aさん：次回キャンペーンの予算は30万円で確定しました。
Bさん：バナー制作はいつまでに進めましょうか？
Aさん：来週金曜日までにデザイン初稿をお願いします。
</会議議事録>
<指示>
以下の元データをもとに、顧客情報のXMLデータを作成してください。
</指示>

<元データ>
・田中健太、28歳、営業部、内線101
・鈴木花子、35歳、総務部、内線202
</元データ>

<出力ルール>
・一番外側は <顧客リスト> タグで囲むこと。
・各メンバーは <顧客> タグで囲むこと。
・部署は属性として持たせること。
</出力ルール>
<指示>
以下の翻訳テキストを、ビジネスメールに適した丁寧な英語に翻訳してください。
</指示>

<固有ルール>
・「製品A」は Product-A と表記してください。
・納期遅延のお詫びが伝わる自然な表現にしてください。
</固有ルール>

<翻訳テキスト>
製品Aの納期についてですが、予定より2日遅れる見込みです。
大変申し訳ございません。
</翻訳テキスト>
⚠️
プロンプト内のタグは「本物のXML」と完全同一でなくてもよいAIへの指示整理が目的なら、読みやすいタグ名を使えば十分です。ただし、閉じタグ忘れや範囲の混乱は避けましょう。

少し発展：知っておくと理解が深まる用語

Well-formed XML と Valid XML

Well-formed は、タグの閉じ方やルート要素など、XMLの基本構文が正しい状態です。

Valid は、DTDやXML Schemaなどで決めた設計図にも合っている状態です。

DTD / XSD：XMLの設計図

「このXMLには <社員> が必要」「<年齢> は数値」といったルールを定義します。業務システム間でデータ品質をそろえる時に役立ちます。

Namespace：同じタグ名の衝突を防ぐ

複数のXML仕様を混ぜると、同じ <title> でも意味が違うことがあります。Namespaceは、タグの所属を明確にする仕組みです。

XPath：XMLの中から場所を指定する

/bookstore/book/title のように、階層をたどって目的の要素を取り出すための指定方法です。Webスクレイピングやデータ抽出でも登場します。

CDATA：タグのような文字をそのまま入れる

プログラムコードや < を含む文章を、そのまま文字データとして扱いたい場合に使うことがあります。

すぐ使えるテンプレート

社員名簿XML

<?xml version="1.0" encoding="UTF-8"?>
<社員名簿>
  <社員 役職="マネージャー">
    <氏名>佐藤 一郎</氏名>
    <部署>営業部</部署>
    <内線番号>1234</内線番号>
  </社員>
  <社員 役職="一般">
    <氏名>鈴木 花子</氏名>
    <部署>総務部</部署>
    <内線番号>5678</内線番号>
  </社員>
</社員名簿>

XMLを書く前・AIに渡す前のチェックリスト

✓
ルート要素は1つか
文書全体を包むタグが1つだけになっている。
✓
開始タグと終了タグは一致しているか
大文字小文字も含めて一致している。
✓
入れ子の順番は正しいか
後から開いたタグを先に閉じている。
✓
属性値は引用符で囲んでいるか
id="001" のように書く。
✓
AIプロンプトでは範囲が明確か
<指示>、<対象テキスト>、<出力形式> を分ける。
