🧩 Claude Agent Skills / Authoring Guide

良いSkillは「短く・見つけやすく・検証できる」

Skillは、Claudeに再利用可能な手順・知識・スクリプトを渡す仕組みです。重要なのは、長い説明ではなく、Claudeが必要な時に正しい情報へ到達できる情報設計です。

最重要メッセージ

SKILL.mdは「全部を書く場所」ではなく、「Claudeが迷わず必要情報に進むための入口」です。詳細は必要時に読ませる構造にします。

1. サマリー：Skill作成の結論

Claudeに“毎回同じ説明をしなくてよい状態”を作る。ただし、詰め込みすぎると逆に性能を落とします。

First Answer
✂️

短く書く

Claudeが既に知っている一般説明は削る。必要な手順・制約・判断基準だけを書く。

🔎

見つけやすくする

nameとdescriptionがSkill選択の入口。何をするか、いつ使うかを具体的に書く。

🗂️

分けて置く

SKILL.mdは概要。詳細・API・事例・スクリプトは別ファイルにして必要時だけ読む。

✅

検証する

評価シナリオを先に作り、実利用で観察し、改善する。想像ではなく挙動で直す。

一言でいうと：Skillは「Claude専用の業務マニュアル」ではなく、「Claudeが迷わないための軽量ナビゲーション＋検証済み手順」です。

2. 全体像：Skillが使われる流れ

Claudeは全Skillを丸ごと読むわけではありません。最初に読み込まれるのはメタデータ中心です。

Architecture
① メタデータname / description が事前に見える
② Skill選択依頼内容に合うSkillをClaudeが判断
③ SKILL.md必要になった時だけ本文を読む
④ 参照ファイル詳細資料・例・APIを必要時に読む
⑤ スクリプト読むより実行。出力だけが文脈を使う
64
name 最大文字数

小文字・数字・ハイフンのみ。reserved wordsは避ける。

1024
description 最大文字数

何をするか、いつ使うかを具体化する。

500
SKILL.md本文の目安行数

超えるなら分割。必要時に読む構造にする。

3. 基本原則：Claudeを過小評価しない

Claudeは多くの一般知識を既に持っています。Skillには、再利用したい固有知識・制約・手順を入れます。

Core Principles

Concise is key

PDFとは何か、ライブラリとは何か、といった一般説明は不要。手順・推奨ツール・注意点だけ残す。

自由度を調整

壊れやすい作業は低自由度。判断が必要な作業は高自由度。作業の危険度で指示粒度を変える。

モデル別にテスト

Haiku / Sonnet / Opus で効き方が変わる。軽量モデルには少し具体性が必要な場合がある。

自由度向いている場面書き方
高コードレビュー、文章改善、分析など複数の正解がある観点・判断基準を示す
中推奨パターンはあるが、設定で変化するテンプレートや疑似コードを示す
低DB移行、破壊的変更、厳密な手順実行コマンド・順序・禁止事項を明記

4. 構造設計：name と description が勝負

ClaudeがSkillを発見できなければ、どれだけ本文が良くても使われません。

Discovery
良いdescription
「Excelファイルを分析し、ピボットテーブルとグラフを作成する。Excel、スプレッドシート、表形式データ、.xlsxの分析時に使用。」
悪いdescription
「データ処理を助ける」「ファイルを扱う」「便利なツール」

命名ルール

  • 小文字・数字・ハイフンのみ
  • 動名詞型が分かりやすい：processing-pdfs
  • helper / utils / files など曖昧語は避ける
  • anthropic / claude など予約語は避ける

descriptionの書き方

  • 三人称で書く
  • 何をするかを具体的に
  • いつ使うかのトリガー語を含める
  • 100個以上のSkillから選ばれる前提で書く
SKILL.md frontmatter 例 
---
name: processing-pdfs
description: Extracts text and tables from PDF files, fills forms, and merges documents. Use when working with PDF files or when the user mentions PDFs, forms, or document extraction.
---

# PDF Processing

## Quick start
Use pdfplumber for text extraction.

5. 段階的開示：SKILL.mdは目次にする

全部を本文に詰め込まず、必要な情報だけをClaudeが読む構造にします。

Progressive Disclosure

推奨構造

pdf/
├── SKILL.md
├── FORMS.md
├── reference.md
├── examples.md
└── scripts/
    ├── analyze_form.py
    ├── fill_form.py
    └── validate.py

設計のコツ

  • SKILL.mdは概要とナビゲーション
  • 詳細は1階層下のファイルへ
  • 長い参照ファイルには目次を付ける
  • ファイル名は内容が分かる名前にする
深すぎる参照がダメな理由
SKILL.md → advanced.md → details.md のように深くすると、Claudeが途中ファイルを一部だけ読んで終わる可能性があります。重要ファイルはSKILL.mdから直接リンクするのが安全です。

6. ワークフローとフィードバックループ

複雑な作業は、Claudeが飛ばさないようにチェックリスト化します。

Workflows

Step 1｜分析

対象ファイルや資料を確認し、必要情報を抽出する。

Step 2｜計画ファイルを作る

変更内容やフィールド対応をJSONなど機械検証しやすい形にする。

Step 3｜検証する

スクリプトやチェックリストで、欠落・不整合・禁止事項を確認する。

Step 4｜実行する

検証が通ってから本処理を行う。

Step 5｜再検証する

出力を確認し、失敗時は計画に戻る。

重要：「Run validator → fix errors → repeat」のループを明示すると、品質が大きく安定します。

7. 共通パターン：迷ったらこの3つ

Skillには、Claudeが出力形式や判断分岐を理解しやすくする型があります。

Patterns

Template pattern

厳密な形式が必要な場合は「必ずこの構造」と明記。柔軟性が必要な場合は「基本形だが判断で調整」と書く。

## Report structure
ALWAYS use this exact structure:
# [Analysis Title]
## Executive summary
## Key findings
## Recommendations

Examples pattern

品質がスタイルに依存する場合は、入力→出力の例を複数示す。説明よりも例の方が伝わりやすい。

Input: Fixed bug where dates displayed incorrectly
Output:
fix(reports): correct date formatting

Use UTC timestamps consistently.

Conditional workflow pattern

作成か編集か、PDFかExcelか、通常処理かOCRか。判断点を明示し、進むべき手順を分岐させる。

1. Determine task type:
   Creating new content? → Creation workflow
   Editing existing content? → Editing workflow

8. 評価と改善：ドキュメントより評価を先に作る

想像で大量の説明を書くのではなく、実際にClaudeが失敗する点を見つけ、それをSkillで補います。

Evaluation

① Gapを確認

Skillなしで代表タスクを実行し、失敗・不足・迷いを記録。

② 評価を作成

最低3つのシナリオを用意。期待動作を明文化。

③ 最小限を書く

評価を通すために必要な指示だけを書く。

Claude A / Claude B の改善サイクル

Claude AでSkillを設計し、Claude Bで実タスクを試す。Bの失敗をAに戻して、指示・構造・強調を改善します。

評価シナリオの例 
{
  "skills": ["pdf-processing"],
  "query": "Extract all text from this PDF file and save it to output.txt",
  "files": ["test-files/document.pdf"],
  "expected_behavior": [
    "Reads all pages",
    "Extracts text without missing pages",
    "Saves output.txt in readable format"
  ]
}

9. アンチパターン：やりがちな失敗

Claudeの性能を落とすのは、情報不足だけではありません。選択肢過多や曖昧さも失敗原因です。

Avoid
避けることなぜダメか代替
Windows風パスscripts\helper.py はUnix系環境で壊れやすいscripts/helper.py
選択肢を出しすぎるClaudeが迷う。品質がブレる標準手段を1つ示し、例外時だけ代替
時限情報を本文に混ぜる時間経過で誤情報になるCurrent method と Old patterns に分ける
用語ゆれAPI endpoint / URL / route が混在すると混乱用語を1つに統一
ツール前提の曖昧さ入っていないパッケージを想定して失敗依存関係とインストール方法を明記

10. 実行コード付きSkill：読ませるより実行させる

決定的な処理、検証、変換は、Claudeに毎回コードを書かせるより、同梱スクリプトを使わせる方が安定します。

Executable Skills

良いスクリプト

  • エラーをClaudeに丸投げしない
  • 例外処理がある
  • 設定値の理由がコメントで分かる
  • 検証結果が具体的

中間成果物を作る

  • plan.json
  • changes.json
  • fields.json
  • validation_report.txt
Plan構造化された変更案
Validateスクリプトで検証
Execute本処理
Verify出力確認
MCPツールを使う場合
ツール名は完全修飾名で書きます。例：BigQuery:bigquery_schema、GitHub:create_issue。複数MCPサーバーがある時に「tool not found」を防げます。

11. 共有前チェックリスト

Skillを配布・運用する前に、最低限ここを確認します。

Checklist
Core quality
  • descriptionが具体的
  • いつ使うかが分かる
  • SKILL.md本文は500行未満
  • 用語が統一されている
  • 例が抽象的でなく具体的
  • 参照は1階層深まで
Code and scripts
  • スクリプトが問題を解決する
  • 例外処理がある
  • 謎の定数がない
  • 依存パッケージを明記
  • 検証・確認ステップがある
  • パスは / を使う
Testing
  • 3つ以上の評価がある
  • Haiku / Sonnet / Opusでテスト
  • 実利用シナリオで確認
  • チーム利用ならフィードバック反映

12. そのまま使えるSkill設計ミニ雛形

まずは小さく始め、評価で足りない部分だけ増やします。

name: analyzing-spreadsheets
description: Analyzes Excel and CSV spreadsheets, summarizes tabular data, detects data quality issues, and creates charts or reports. Use when working with .xlsx, .csv, spreadsheets, tables, or business data analysis.

# Spreadsheet Analysis

## Quick start
1. Inspect the workbook structure.
2. Identify sheets, columns, row counts, and missing values.
3. Summarize key findings before creating charts or reports.

## Default workflow
- [ ] Load file and inspect schema
- [ ] Validate required columns
- [ ] Clean obvious formatting issues
- [ ] Analyze requested metric
- [ ] Create output report
- [ ] Verify totals and labels

## References
**Pivot table patterns**: See [reference/pivot-tables.md](reference/pivot-tables.md)
**Chart rules**: See [reference/charts.md](reference/charts.md)
**Validation scripts**: Run `python scripts/validate_spreadsheet.py input.xlsx`

## Rules
- Use forward slashes in file paths.
- Validate calculations before final output.
- If columns are ambiguous, inspect headers and sample rows before assuming meanings.
