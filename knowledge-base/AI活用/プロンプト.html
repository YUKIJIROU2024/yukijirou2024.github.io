Claudeへの指示は「曖昧なお願い」ではなく、
役割・文脈・出力形式・判断基準を渡す設計図にする。

このページの要点は、Claudeの最新モデルを使うときに、プロンプトをどう書けば安定した品質・形式・行動を得られるか、という実務ガイドです。特に重要なのは、明確な指示、XML構造化、例示、長文入力の配置、ツール利用の明示、thinking / effort の調整、そしてエージェント作業での状態管理です。

出典：Anthropic Claude API Docs「Prompting best practices」（2026年6月8日に確認）

覚え方：新人に業務を頼むつもりで書く

Claudeは高性能でも、あなたの職場の前提・評価基準・好みは知りません。「何を」「なぜ」「どの順番で」「どの形式で」「どこまでやるか」を具体化すると安定します。

3基本原則：まず押さえる5つ

🎯

明確・直接

期待する出力、制約、順番をはっきり書く。曖昧な「いい感じに」は避ける。

🧭

文脈を渡す

目的や背景を説明すると、Claudeは判断の優先順位を合わせやすい。

🧪

例を入れる

出力形式・トーン・分類基準は、3〜5個の良い例で安定しやすい。

🏷️

XMLで分ける

instructions / context / input / examples のようにタグで区切る。

👤

役割を与える

「Python専門の支援者」「金融実務に詳しいレビュアー」など一文でも効く。

📚

長文は上、質問は下

長い資料は先に置き、最後に依頼を書く。複数文書はメタデータ付きで構造化。

悪い依頼と良い依頼の違い
観点弱いプロンプト強いプロンプト
依頼この資料をまとめて役員向けに、結論→根拠→リスク→次アクションの順で400字以内に要約
形式わかりやすく表を1つ、重要ポイントを3つ、最後に判断材料を箇条書き
文脈改善案を出して目的は採用応募率の改善。対象は未経験のパート希望者。専門用語を避ける
例いい感じのトーンで以下の例と同じ文体・見出し構成で出力

4出力制御：欲しい形を先に決める

形式を安定させるコツ

Do「何をするか」で書く。例：「滑らかな文章で書く」「表形式で出す」。

XML出力の領域をタグで指定する。

Style matchプロンプト自体の書き方を、期待する出力形式に近づける。

避けたい書き方

「Markdownを使わないで」のような否定形だけでは弱い場合があります。代わりに「自然な段落で書く」「番号付きリストは必要時のみ」など、望む形を明示します。

サンプル：レポートを長文になりすぎず整える
<writing_style> 結論を最初に書く。 文章は短く区切る。 箇条書きは、比較・手順・チェックリストに限って使う。 強調すべきリスクと次アクションを最後に示す。 </writing_style>
LaTeXや数式表現を避けたい場合

数式を普通の文字で書いてほしい場合は、「LaTeX、MathJax、\( \)、$、\frac{}{} は使わない」と明記します。

5ツール利用：提案だけか、実行までかを明示

実行してほしい場合

「修正案を出して」だと提案で止まることがあります。ファイル編集、検索、テスト実行などの具体的行動をしてほしいなら「実装して」「確認して」「変更して」と書きます。

慎重にしてほしい場合

削除、force push、外部投稿、共有システム変更など、戻しにくい行為は事前確認させる指示を入れます。

ツール利用プロンプトの型
<default_to_action> ユーザーの意図が実装・修正・確認に見える場合は、提案だけで終わらず、必要なファイル確認やテストを行って実行してください。 ただし、削除・外部送信・共有環境への変更など、戻しにくい操作は事前に確認してください。 </default_to_action>
並列ツール利用の考え方

独立した検索・ファイル確認・複数資料読み込みは並列化すると速い。一方、前の結果が次の入力に必要な作業は順番に行わせます。

6思考・推論：effort と adaptive thinking を使い分ける

low

短い分類、定型文、低コスト・低遅延向け。

medium

コストを抑えつつ、ある程度の推論が必要な作業。

high

多くの知的作業の基準。調査、設計、レビューに向く。

xhigh / max

長期エージェント、複雑なコーディング、大規模調査向け。

実務上の見立て

「答えが浅い」と感じたときは、プロンプトで無理に長く考えさせるより、effort を high / xhigh に上げるほうが本筋です。逆に、考えすぎ・調べすぎが出るなら effort を下げるか、使うべき場面を限定します。

adaptive thinking の使いどころ

多段階ツール利用、複雑なコード修正、長期エージェント作業などでは adaptive thinking が有効です。簡単な質問では直接回答し、難しい場面で深く考えるように調整できます。

thinking={"type":"adaptive"} output_config={"effort":"high"}

7エージェント運用：長い仕事は状態管理が命

状態を残す

progress.txt、tests.json、git log などで、作業状況を外部化する。

テストを先に作る

長期作業では、テストが進捗と品質の基準になる。

増やしすぎない

サブエージェントやファイル作成は便利だが、過剰利用は避ける。

長期作業の開始プロンプト例
この作業は長期タスクです。 まず作業計画、検証方法、進捗管理ファイルを作ってください。 作業は小さな単位で進め、各ステップの完了条件を明確にしてください。 削除・外部公開・共有環境への変更など、戻しにくい操作は確認してください。
調査タスクの型
調査の成功条件を最初に定義してください。 複数ソースで検証してください。 仮説を複数立て、確信度を更新しながら進めてください。 最後に、結論・根拠・不確実な点・次に確認すべきことを分けて示してください。

8コーディング：実装力を引き出す注意点

レビューは「見つけたものを全部出す」段階と「重要度で絞る」段階を分ける

最初から「重大なものだけ」と指示すると、低〜中リスクの実バグを落とすことがあります。発見段階では網羅性を優先し、信頼度と深刻度を併記させます。

過剰な抽象化・余計なファイル作成を抑える

「求められた範囲だけ」「一回限りの処理にヘルパーを作らない」「将来の仮定に備えすぎない」と明記します。

開いていないコードについて断言させない

特定ファイルに言及する場合は、回答前に必ず対象ファイルを読むよう指示します。憶測の説明を減らせます。

テストを通すだけのハードコードを避ける

「テストは仕様確認の手段であり、特定ケースだけに合わせない」と明記します。一般解を求める姿勢を促します。

9移行時の注意：新モデルでは“強すぎる指示”を見直す

Claude 4.6以降の傾向

指示追従が強く、ツールやスキルの呼び出しも以前より適切に動くため、昔の「必ず使え」「迷ったら使え」のような強い指示は過剰動作につながることがあります。

prefill からの移行

Claude 4.6以降では最後のassistantターンに事前入力する方式が使えないケースがあります。出力形式は、プロンプト内の明示的な形式指定や例示で制御します。

期待する出力を具体化したか
thinking / effort の設定を見直したか
過剰なツール利用指示を弱めたか
アニメーションやUI要素は明示したか
前提・制約・評価基準を最初に入れたか
長文入力は上、質問は下にしたか

10すぐ使えるプロンプト型

汎用テンプレート

<role> あなたは［専門領域］に詳しい実務支援者です。 </role> <context> 目的：［何のために使うか］ 読み手：［誰が読むか］ 制約：［文字数・トーン・禁止事項］ </context> <task> 次の入力をもとに、［成果物］を作成してください。 </task> <output_format> 1. 結論 2. 根拠 3. 注意点 4. 次アクション </output_format>

長文資料要約テンプレート

<documents> <document index="1" source="資料A"> <document_content> ［ここに本文］ </document_content> </document> </documents> <instructions> まず関連箇所を短く引用してから、 結論・論点・リスク・実務対応に分けて要約してください。 </instructions>

コードレビュー用

発見段階では、重要度で絞らず、見つけた問題をすべて報告してください。 各項目に以下を付けてください。 - 問題の場所 - 何が起きるか - 信頼度 - 推定深刻度 - 修正方針 その後、優先順位順に並べ替えてください。

デザイン生成用

一般的なAI風デザインを避けてください。 この用途に合う独自の視覚テーマを3案出し、 背景色・アクセント色・フォント方向性・雰囲気を比較してください。 選定後、その方向だけで実装してください。
